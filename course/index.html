01

What This App Does

A government office mail tracking system that knows exactly where every piece of mail is, who's handling it, and how long it's been sitting there.

Imagine a Busy Government Office

Papers arrive every day — letters, memos, requests, reports. They need to go to the right person, get processed, and move along. But without tracking, mail gets lost, delayed, or forgotten.

💡
The Core Purpose

This system answers one critical question: "Where is this mail right now, and who's responsible for it?" It tracks every handoff, every status change, and how long each step takes.

Four Types of Users

Not everyone can do everything. The system has a role-based permission system that mirrors real office hierarchy.

👑 AG
Accountant General
  • Create mail for any section
  • View everything
  • Reassign anyone
  • Close any mail
📋 DAG
Deputy AG
  • Create for their section
  • View section scope
  • Reassign within section
📝 SrAO/AAO
Staff Officers
  • Work on assigned mail
  • Update status
  • Close their own items
  • Own subsection scope
📎 Clerk/Auditor
Support Staff
  • Create within scope
  • View assigned scope
  • Limited by subsection

The Four Stages of Mail

Every piece of mail flows through these stages automatically. The system tracks time spent at each step — crucial for identifying bottlenecks.

📥 Created 👤 Assigned ⚡ In Progress ✅ Closed
1
Created

Someone logs a new piece of mail with details: where it came from, what it's about, when it was received.

2
Assigned

The mail gets routed to an officer who becomes responsible. The system tracks who has it and when it was handed off.

3
In Progress

The assigned officer starts working. They can add remarks, upload PDFs, or reassign to someone else.

4
Closed

Work complete! The mail is marked done with completion date and final remarks recorded.

Meet the Database

Three core models that store everything: Users, Mail Records, and an Audit Trail that remembers every action.

The User Model

Every person in the system is a User with a role that determines what they can see and do. Think of it like employee badges with different clearance levels.

CODE 
class User(AbstractUser):
    role = models.CharField(
        choices=ROLE_CHOICES
    )
    sections = models.ManyToManyField(
        Section, blank=True
    )
    subsection = models.ForeignKey(
        Subsection,
        null=True,
        blank=True
    )
PLAIN ENGLISH

Each user has a role (AG, DAG, SrAO, etc.) that determines their permissions.

DAGs can manage multiple sections — stored as a list they belong to.

SrAOs and AAOs belong to one specific subsection (like a team).

🗝️
Many-to-Many vs ForeignKey

A DAG can manage many sections, and a section can have many DAGs — that's a many-to-many relationship. But a SrAO belongs to one subsection — that's a ForeignKey.

The MailRecord Model

This is the heart of the system. Every piece of tracked mail is a MailRecord with fields for tracking, assignment, and status.

CODE 
class MailRecord(models.Model):
    sl_no = models.CharField(
        max_length=10,
        unique=True
    )
    letter_no = models.CharField(
        max_length=200
    )
    assigned_to = models.ForeignKey(
        User,
        related_name='assigned_mails'
    )
    current_handler = models.ForeignKey(
        User,
        related_name='current_mails'
    )
    status = models.CharField(
        choices=STATUS_CHOICES
    )
PLAIN ENGLISH

sl_no: Serial number like "2026/001" — auto-generated each year.

letter_no: The actual reference number from the physical letter.

assigned_to: Who was originally given this mail.

current_handler: Who has it RIGHT NOW (can change via reassignment).

status: Created → Assigned → In Progress → Closed.

The Audit Trail — Your Safety Net

Every action gets logged. Who did what, when, and what changed. This creates accountability and makes it possible to reconstruct exactly what happened.

CODE 
class AuditTrail(models.Model):
    mail_record = models.ForeignKey(
        MailRecord
    )
    action = models.CharField(
        choices=ACTION_CHOICES
    )
    performed_by = models.ForeignKey(
        User
    )
    timestamp = models.DateTimeField(
        auto_now_add=True
    )
    remarks = models.TextField()
    old_value = models.JSONField()
    new_value = models.JSONField()
PLAIN ENGLISH

Links to the specific mail record being modified.

What happened: CREATE, ASSIGN, REASSIGN, CLOSE, etc.

Who performed the action — linked to their user account.

When it happened — automatically set by the system.

Any comments they added explaining their action.

Before/after values for tracking what actually changed.

📜
Immutable Audit Logs

Audit records can never be edited or deleted. This is crucial for accountability — like writing in pen instead of pencil. The system enforces this at the database level.

API & Permissions

How the backend exposes data securely, checks permissions on every request, and uses JWT tokens to verify who's asking.

The API Layer

The frontend talks to the backend through an API. Think of it as a menu of operations: create mail, get mail list, reassign, close, etc.

📡
REST Endpoints

Standard HTTP methods: POST to create, GET to read, PUT to update, DELETE to remove.

🔐
JWT Authentication

JSON Web Tokens prove identity. Like a concert wristband that shows you're allowed in.

🛡️
Permission Classes

Every endpoint checks: "Is this user allowed to do this?" Role determines access.

How JWT Authentication Works

When you log in, the server gives you two tokens: an access token (short-lived, like a daily pass) and a refresh token (longer-lived, like a season pass).

api.interceptors.request.use(
    (config) => {
        const token = localStorage
            .getItem('access_token');
        if (token) {

PLAIN ENGLISH

Before every API request, we automatically attach the stored token.

Get the token from browser storage where it was saved after login.

Add it as a Bearer token in the Authorization header — like showing your ID.

The server checks this token to verify who you are.

Token Refresh: The Silent Renewal

Access tokens expire after 24 hours. Instead of forcing you to log in again, the system silently refreshes them using the longer-lived refresh token.

CODE 
if (error.response?.status === 401
    && !originalRequest._retry) {
    if (isRefreshing) {
        # Queue requests until refresh done
        return new Promise((resolve) => {
            failedQueue.push({ resolve });
        });
    }
    originalRequest._retry = true;
    isRefreshing = true;
    const newToken = await refreshAccessToken();
    processQueue(null, newToken);
}
PLAIN ENGLISH

If a request fails with 401 (unauthorized), the token might be expired.

If a refresh is already in progress, queue this request to retry after.

Mark this request as retrying so we don't loop infinitely.

Ask the server for a new access token using the refresh token.

Retry all queued requests with the new token.

Frontend Architecture

How the React interface is organized: pages, components, services, and the auth context that remembers who you are.

The Frontend Structure

React breaks the UI into reusable pieces. Here's how the Mail Tracker organizes its code:

frontend/src/
pages/ Full-page components (Login, MailList, CreateMail)
components/ Reusable pieces (dialogs, buttons, forms)
services/ API calls organized by domain
context/ Global state (AuthContext)
layouts/ Page shells with navigation

AuthContext: The Memory Bank

React's Context API lets any component check "Who's logged in?" without passing that data through every intermediate component.

CODE 
const AuthContext = createContext(null);

export const AuthProvider = ({ children }) => {
    const [user, setUser] = useState(null);

    const login = async (username, password) => {
        const { user } = await
            authService.login(username, password);
        setUser(user);
    };

    return (
        <AuthContext.Provider value={{ user, login }}>
            {children}
        </AuthContext.Provider>
    );
};
PLAIN ENGLISH

Create a context container to hold authentication state.

The Provider wraps the app and manages the user data.

Store the current user in state — null means not logged in.

Login function calls the API and saves the user data.

Make user and login function available to all child components.

🧠
State Management Pattern

Instead of each page fetching user data, AuthContext fetches it once at startup and shares it everywhere. This prevents flickering and redundant API calls.

Services: API Organization

All API calls live in services/ — one file per domain. This keeps components clean and makes API changes easy to find.

Security Deep Dive

How the backend enforces role-based access. Permission classes, scoped queries, and why the UI is not the authority.

Permission Classes

Every API endpoint has a permission class that runs before the main logic. If permission check fails, the request is rejected immediately.

CODE 
class IsAGOrDAG(BasePermission):
    def has_permission(self, request, view):
        user = request.user
        return user.is_authenticated and (
            user.is_ag() or
            user.is_dag()
        )

class MailRecordViewSet(ModelViewSet):
    permission_classes = [IsAGOrDAG]

    def create(self, request):
        # Only runs if IsAGOrDAG passes
PLAIN ENGLISH

Define a custom permission rule: user must be AG or DAG.

Get the user from the request (set by JWT authentication).

Return True only if authenticated AND has AG or DAG role.

Apply this permission to the entire MailRecord viewset.

Create action only runs if permission check passed.

⚠️
Backend is the Authority

The frontend can hide buttons, but the backend must enforce. A malicious user could craft API requests directly. Never trust the frontend alone — always verify permissions server-side.

Scoped Queries: What You See

Different users see different records. AG sees everything. DAG sees their sections. SrAO sees only their subsection.

CODE 
def get_queryset(self):
    user = self.request.user

    if user.is_ag():
        return MailRecord.objects.all()

    elif user.is_dag():
        section_ids = user.sections.values_list(
            'id', flat=True)
        return MailRecord.objects.filter(
            section_id__in=section_ids)

    else:
        return MailRecord.objects.filter(
            subsection=user.subsection)
PLAIN ENGLISH

Get the currently logged-in user from the request.

If user is AG: return ALL mail records, no filtering.

If DAG: get their managed section IDs.

Return only mail records in those sections.

For everyone else: return only mail in their subsection.

Security Checklist

CSRF Cross-Site Request Forgery protection enabled for state-changing operations
CORS Cross-Origin Resource Sharing configured to only allow trusted frontend domains
SQL Injection Protected by Django ORM — queries are parameterized, never concatenated
XSS Cross-Site Scripting protection via Django's auto-escaping templates
Password Hashing Passwords hashed with PBKDF2 — never stored in plain text
JWT Expiry Access tokens expire after 24 hours, refresh tokens after 7 days
